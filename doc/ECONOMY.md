# Economy System

상인 NPC가 아이템을 사고팔기 위한 화폐 시스템과 미감정 아이템 시스템 설계. 2026-06-10 논의 결과를 정리한 문서이며, 아직 구현 전이다.

## 레퍼런스 조사

### D&D 5e: 다층 화폐

5종류 동전을 십진법 비율로 사용한다.

| 동전 | 약자 | 가치 |
|------|------|------|
| 백금화 (Platinum) | pp | 10 gp |
| 금화 (Gold)       | gp | 기준 단위 |
| 호박금화 (Electrum) | ep | 0.5 gp |
| 은화 (Silver)     | sp | 0.1 gp |
| 동화 (Copper)     | cp | 0.01 gp |

- 동화/은화는 일상 경제(맥주 4cp, 서민 하루 생활비 ~2sp), 금화는 모험가 경제(롱소드 15gp, 판금갑옷 1,500gp)로 층이 나뉜다.
- 동전에 무게가 있어(50닢 = 1파운드) 돈 자체가 운반 문제가 된다.
- 거래 기본 룰: 정가에 사고, 팔 때는 반값.
- 옛 판본(AD&D)은 1gp = 20sp = 200cp 같은 비십진 비율이었으나 번거로워서 5e에서 십진법으로 정리됐다.

### NetHack: 단일 화폐

- 금화(zorkmid, `$`) 하나뿐. 은/동은 화폐가 아니라 아이템 재질로만 존재한다.
- 구매 가격은 카리스마에 따라 변동 — CHA가 높으면 정가의 절반, 낮으면 2배까지.
- 판매 가격은 기본가의 절반 수준 (사고팔기 차익 불가).
- 모든 아이템에 고정 기본가가 있어, 미감정 물약/두루마리를 상점 가격으로 역추적하는 가격 감정(price identification) 테크닉이 유명하다.
- 외상(credit), 빚, 절도 시 경찰(Kops) 출동 같은 부가 시스템도 있다.

### MMO 관례

WoW는 내부적으로 구리 단위 정수 하나로만 저장하고, 1골드 = 100실버 = 10,000코퍼는 순수하게 클라이언트 표시 포맷이다.

## 결정: 화폐

**단일 화폐**로 간다 (NetHack/WoW 방식).

1. 서버는 최소 단위 정수 하나로 저장한다. 큰 거래/누적을 대비해 64비트 정수를 쓴다.
2. 금/은/동 표기는 클라이언트 표시 계층에서만 변환한다 (비율은 1골드 = 100실버 = 10,000코퍼 예정, 구현 시 확정).
3. 동전을 별도 아이템/인벤토리 오브젝트로 만들지 않는다. D&D식 다층 화폐는 환전·무게·거래 UI 복잡도만 늘리고 온라인 게임에서 얻는 재미가 없다.

## 결정: 상인 거래

- **구매는 정가, 판매는 기본가의 30~50%.** D&D와 NetHack이 공통으로 쓰는 스프레드 패턴으로, 사고팔기 무한 차익을 막는 최소 장치다.
- 상인이 **골드 싱크** 역할을 하도록 설계한다 (감정 수수료, 수리비 등으로 돈을 경제에서 제거). 멀티플레이어 인플레이션 대응의 핵심 통로.
- CHA 연동: [COMBAT.md](COMBAT.md)에서 CHA가 "NPC 반응, 거래"를 담당하므로, NetHack식 카리스마 가격 보정(CHA 높으면 할인, 낮으면 할증)을 자연스럽게 얹을 수 있다. 히든 클래스 Merchant의 CHA +3과도 맞물린다.

## 악용 방지: 캐릭터 재활용 머니 펌프

캐릭터 생성이 환금 가능한 가치를 공짜로 발행하면, "캐릭터 생성 → 가치 이전 → 삭제 → 반복" 루프가 무한 골드 파우셋이 된다. 이전 경로는 이미 존재한다(바닥 드랍 → 다른 캐릭터 줍기). 상인 스프레드(판매 40%)는 펌프를 느리게 할 뿐 막지 못한다 — 발행 자체를 끊어야 한다.

**원칙: 캐릭터 생성 시점에 주는 것은 환금 불가여야 한다.** (2026-06-10 적용 완료)

- **시작 골드는 지급하지 않는다** (gold = 0). 첫 수입은 몬스터 드랍 판매 등 플레이에서 나온다.
- **시작 장비는 `basePrice` 없는 전용 정의로 준다**: 신규 캐릭터는 Worn Iron Sword(`worn_iron_sword`, iron_sword와 스탯 동일·가격 없음)를 장착하고 시작한다. 서버가 가격 없는 아이템의 매입을 거부하므로 추가 인프라 없이 환금이 막힌다. 캐릭터 귀속(bound, 드랍·거래 불가) 플래그는 미감정 아이템(Phase 4)의 인스턴스 시스템과 함께 도입 검토.
- 보조 마찰(캐릭터 생성/삭제 쿨다운, 저레벨 드랍·판매 제한)은 변종 악용이 관찰되면 추가한다.

## LLM NPC 거래와 흥정 (Rica, Karl)

NPC는 LLM이 조종한다 ([AGENT_CLIENT.md](AGENT_CLIENT.md), 현재 상인 Rica와 경비병 Karl). 이 특징을 살려 **LLM이 대화 맥락에 따라 할인/할증을 결정하는 흥정 시스템**을 얹고, 상인뿐 아니라 **모든 LLM NPC와 거래가 가능**하게 한다. "말로 설득할 수 있는 상인"과 "필요한 물건을 사러 다니는 주민"은 LLM 구동 NPC만 가능한 차별점이다.

### 원칙: LLM이 제안하고, 서버가 집행한다

플레이어는 반드시 LLM을 구슬리거나 프롬프트 인젝션으로 폭탄 할인을 뜯어내려 시도한다 (실사례: 쉐보레 딜러 챗봇이 1달러에 차를 팔기로 약속한 사건). **LLM은 결국 뚫린다고 가정하고, 방어는 전부 서버에서 한다.**

1. **서버가 가격 밴드를 계산한다.** 기본가 × 스프레드 × CHA 보정 등으로 `[최소가, 최대가]` 밴드를 산출한다 (예: 정가 ±20%).
2. **LLM은 밴드 안에서 보정치만 고른다.** 절대 가격이 아니라 구조화된 액션(예: `offer_deal(item, modifier_pct, reason)`)으로 할인율을 제안하고, 서버가 밴드를 벗어나면 클램프하거나 거부한다.
3. **차익 불변식(invariant)을 서버가 보장한다.** 최대 할인 적용 구매가 > 최대 판매가가 항상 성립해야 한다 (예: 구매 최저 80% > 판매 최대 50%). LLM이 어떤 결정을 해도 사고팔기 머니 펌프가 생길 수 없는 구조.
4. **예산과 쿨다운.** 플레이어별 일일 할인 한도, NPC별 일일 할인 장부(Rica의 "지갑")를 둔다. 흥정 시도 자체에도 쿨다운을 둔다 — 어뷰징 방지 겸 LLM 호출 비용 통제.
5. **전 결정 로깅.** LLM의 모든 가격 결정을 사유(reason)와 함께 기록한다. 익스플로잇 탐지와 페르소나 튜닝 양쪽에 필요하다.

### 상인 vs 비상인 NPC

모든 LLM NPC가 거래할 수 있지만, 역할에 따라 공급 모델이 다르다.

| | 상인 (Rica) | 비상인 (Karl 등) |
|---|------------|-----------------|
| 재고 | 무제한 공급 (카탈로그 데이터로 정의) | 실제 인벤토리만 |
| 취급 품목 | 카탈로그 전체 | 본인이 필요로 하는 한두 가지 (위시리스트) |
| 지갑 | 사실상 무제한 | 유한 (급여로 충전) |
| 가격 밴드 | 좁음 (±20%) — 경제의 가격 앵커 | 넓음 — 정말 필요한 물건엔 프리미엄 허용 |
| 역할 | 골드 싱크, 기준 가격 제공 | 롤플레이, 자연 발생 퀘스트 |

- **상인**: 취급 품목은 카탈로그 데이터로 정의한다. 기존 게임 데이터 파이프라인([data-src/](../data-src/)의 CSV에서 작성 → [data/](../data/)의 JSON으로 생성)을 따른다. 플레이어가 판 아이템은 소멸하고 재고는 무제한이다.
- **비상인**: 유한 지갑과 실제 인벤토리 덕분에 거래 규모가 자연히 캡 된다 — LLM이 뚫려도 최대 손실 = NPC 지갑 잔고. 위시리스트(예: Karl이라면 숫돌, 맥주)는 NPC 데이터에 정의하거나 LLM이 화이트리스트 안에서 고르게 한다. 거래 자체는 LLM이 페르소나에 맞게 롤플레잉한다.
- **NPC 급여**: 비상인 지갑의 골드 출처는 서버가 주기적으로 지급하는 급여다. 통제된 faucet이라 인플레 영향이 계산 가능하고, "월급 받아 필요한 물건을 사는 주민"이라는 생활 경제 연출이 된다.
- **차익 불변식의 예외**: 상인↔비상인 간 차익(상인에게 싸게 사서 Karl에게 프리미엄으로 판다)은 의도된 콘텐츠다 — 사실상 자연 발생 배달 퀘스트의 보상. 규모가 NPC 지갑(=급여 누적)으로 캡 되므로 폭주하지 않는다.

### 재미 요소

밴드가 안전을 보장하므로, 밴드 안에서는 LLM이 마음껏 놀게 둘 수 있다:

- **흥정 = 대화 콘텐츠.** 좋은 이야기, 소문, 아부에 Rica가 반응해 할인해주는 것 자체가 롤플레이 콘텐츠. 무례하게 굴면 할증도 가능 (대칭적이고, 페르소나에 부합).
- **CHA 연동.** CHA가 높을수록 서버가 산출하는 밴드 폭이 넓어진다 — NetHack 카리스마 가격 보정의 LLM 버전.
- **단골/평판.** Rica가 단골을 기억하고 (페르소나 메모리) 밴드 안에서 우대.
- **오늘의 품목.** "버섯이 필요하다는 소문을 들었어" — LLM이 화이트리스트 안에서 오늘의 매입 우대 품목을 골라 미니 이벤트를 만든다. 매입가 보너스도 당연히 밴드 제한.
- **자연 발생 퀘스트.** 비상인 NPC가 위시리스트 기반으로 "치유 물약을 구해다 주면 값을 잘 쳐주지" 같은 의뢰를 대화에서 즉석으로 만든다. 별도 퀘스트 시스템 없이 LLM 롤플레잉만으로 fetch 퀘스트가 생긴다.

### 거래 진입 UI

NPC 종류에 따라 클릭의 기본 액션을 다르게 한다. 경비병을 클릭했는데 거래창이 바로 뜨는 것은 어색하다.

- **클릭 = NPC별 기본 액션.** 상인 클릭은 곧바로 거래창 (플레이어 의도의 대부분이 거래). 비상인 클릭은 대화 시작.
- **컨텍스트 메뉴 = 명시적 진입로.** 우클릭(또는 길게 누르기)으로 대화·거래 등의 옵션 메뉴를 띄운다. 메뉴 항목은 하드코딩하지 않고 NPC 데이터에 정의된 상호작용 능력(capability) 목록으로 구성한다 — 퀘스트 시스템이 생기면 항목만 추가되는 구조.
- **LLM이 거래창을 연다.** 대화 중 거래 얘기가 나오면 NPC(LLM)가 `open_trade` 액션을 호출해 거래창을 띄운다. 비상인 거래는 이쪽이 주 진입로 — 경비병과의 거래는 메뉴에서 고르는 것보다 "대화하다가 성사되는" 쪽이 자연스럽고, 위시리스트를 대화로 발견하는 흐름과도 맞는다.

### 구현 메모

- agent-client의 LLM 액션([driver/action.rs](../agent-client/src/driver/action.rs))에 흥정 액션을 추가하고, 서버에 거래 오퍼 검증(밴드 클램프, 불변식, 예산/지갑)을 둔다.
- 거래 검증은 서버가 하므로 상인 카탈로그, 기본가, 비상인 위시리스트는 서버가 읽는 게임 데이터([data/](../data/))에 둔다. agent-client 쪽 페르소나 프롬프트에는 같은 정보를 롤플레이용으로 주입한다.
- 아이템 기본가(`basePrice`) 추가가 선행 작업이다 — 스프레드, 가격 밴드, 가격 감정 전부 기본가에서 파생된다. [data/items.json](../data/items.json)은 [data-src/items.csv](../data-src/items.csv)에서 생성되므로, CSV에 `basePrice` 컬럼을 추가하고 재생성한다.
- 플레이어 채팅은 신뢰할 수 없는 입력으로 취급한다. 페르소나 프롬프트에 방어 지시를 넣되, 실질 방어선은 서버 검증이다.

## 미감정 아이템 (Unidentified Items)

### 문제: MMO에서 NetHack식 미감정은 깨진다

NetHack의 미감정은 **종류(type) 레벨의 비밀**이다 — "소용돌이치는 물약 = 치유 물약"이라는 매핑이 비밀. 이런 비밀은 한 명만 알아내면 위키에 올라가고 끝나서, 영속 서버에서는 수명이 하루다. 미감정이 MMO에서 살아남으려면 비밀을 **개체(instance) 레벨**로 옮겨야 한다.

### 검토한 대안

| 방식 | 내용 | 판정 |
|------|------|------|
| 플레이어별 매핑 | 미감정 외형-효과 매핑을 플레이어마다 다르게 랜덤화 | **기각** |
| 서버/시즌별 매핑 | 매핑을 서버 단위 또는 시즌마다 재랜덤화 | **기각** |
| 디아블로 스타일 | 아이템 종류는 공개, 개체별 랜덤 옵션만 감정 전 숨김 | **채택** |

**플레이어별 매핑 기각 사유:**
- 거래가 망가진다. 같은 아이템이 플레이어마다 다른 이름으로 보이면 거래 채팅이 성립하지 않고, 거래 후 감정 상태 승계도 애매하다.
- 상점 가격으로 정보가 샌다. NetHack식 가격 감정이 역으로 작동해 매핑이 풀린다.
- 클라이언트가 진짜 타입 ID를 받는 순간 애드온/패킷 분석으로 자동 번역된다. 막으려면 와이어 레벨에서 플레이어별 타입 ID 난독화가 필요한데 비용 대비 효용이 없다.

**서버/시즌별 매핑 기각 사유:** 어차피 커뮤니티가 며칠 내에 집단 공략으로 풀어버리며, 들이는 노력에 비해 유저 불편만 커지고 얻는 이득이 적다. 시즌 인프라가 생긴 뒤 소비 아이템 한정으로 재검토할 수는 있다 (시즌 초 집단 퍼즐 풀이 자체를 콘텐츠로 보는 관점, PoE 리그 사례).

### 채택: 디아블로 스타일

- 아이템 종류(기본 이름)는 항상 공개하고, **개체별 랜덤 옵션(affix)을 감정 전까지 숨긴다.** 개체마다 새로 굴리므로 위키 스포일이 원천적으로 불가능하다.
- **서버는 감정 전 옵션 데이터를 클라이언트에 아예 전송하지 않는다.** 데이터마이닝·애드온 노출을 원천 차단하며, 구현도 "필드 숨기기" 수준으로 단순하다.
- 감정 수단은 감정 주문서 구매 또는 상인 NPC 수수료 감정으로, 위의 골드 싱크와 연결한다.

### 확장 아이디어 (추후)

- **미감정 상태 거래 허용**: "안 까본 아이템"이 그 자체로 상품이 되는 플레이어 간 갬블 시장 (디아블로 2의 미감정 유니크 거래 사례).
- **감정을 플레이어 스킬/직업으로**: 감정 숙련도를 두면 "감정사" 역할이 생긴다 (울티마 온라인 사례). 사회적 상호작용 콘텐츠.
- 시즌제 도입 시 소비 아이템 한정 시즌별 외형-효과 매핑 재랜덤화 (현재는 보류).

## 구현 단계

1. **화폐 + 상인 고정가 거래 (MVP)**: `basePrice` 데이터, 플레이어 골드(영속화), 거래 프로토콜, 상인 카탈로그, 거래창 UI. 흥정 없이 정가 구매 / 스프레드 판매. — **구현 완료 (2026-06-10)**, 상세는 [devlog](devlog/2026-06-10.md) 참고.
2. **LLM 흥정**: 가격 밴드 계산, 흥정 액션(`offer_deal`), 예산/쿨다운, 결정 로깅. — **구현 완료 (2026-06-11)**, 상세는 [devlog](devlog/2026-06-11.md) 참고. CHA 연동 밴드(±5~25%p), 게임일 단위 예산(상인 1g / 플레이어 40s), 30초 쿨다운, 딜은 1회용·5분 만료.
3. **비상인 거래**: NPC 지갑/급여, 위시리스트, 컨텍스트 메뉴, `open_trade` 액션.
4. **미감정 아이템**: 아이템 인스턴스별 랜덤 옵션 시스템과 함께 도입.
